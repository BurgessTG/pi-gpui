# Research: Smooth pan/zoom/rendering patterns for infinite canvas / node editors

## Summary
Smooth editors keep camera movement cheap, render only viewport-relevant content, and isolate expensive work behind caches, layers, batches, and zoom-level-of-detail. For a native GPUI/Rust implementation, model the canvas as a camera-independent scene plus screen-space overlay layers: query visible nodes/edges from an incremental spatial index, keep fixed-size controls/nodes in inverse-zoom space, and instrument every interaction with frame-time/culling/cache counters.

## Findings
1. **Use spatial-indexed, exception-aware viewport culling** — tldraw’s current spatial index is an R-tree/RBush manager with per-page rebuilds, bulk loading, and incremental upsert/remove processing from store diffs; culling then queries `getShapeIdsInsideBounds(viewport)` instead of recomputing all candidates from scratch. Its visibility docs also keep selected/editing shapes rendered and let shape types opt out via `canCull`, which is the right model for text measurement, animations, and oversized effects. React Flow’s `onlyRenderVisibleElements` is useful but documented as having overhead, and xyflow’s RFC calls out its current naive AABB checks and points toward BVH/user-supplied spatial queries. **GPUI pattern:** maintain an `rstar`/R-tree per page/layer for node bounds and edge bounds, query an expanded viewport with hysteresis, then force-include selected, editing, dragging, connecting, and unmeasured items. [tldraw SpatialIndexManager](https://github.com/tldraw/tldraw/blob/f69220b8/packages/editor/src/lib/editor/managers/SpatialIndexManager/SpatialIndexManager.ts), [tldraw visibility docs](https://github.com/tldraw/tldraw/blob/main/apps/docs/content/sdk-features/visibility.mdx), [React Flow API](https://reactflow.dev/api-reference/react-flow), [xyflow spatial-query RFC](https://github.com/xyflow/xyflow/issues/4239)

2. **Avoid O(N) viewport work on every pan/zoom** — Excalidraw’s renderer currently iterates `scene.getNonDeletedElements()` and runs `isElementInViewport()` for each element before returning `visibleElements`; a related issue explicitly notes this makes pan/zoom scale linearly with total scene size. That is a useful warning for native code: even if drawing is fast, full-scene traversal can dominate interaction frames. **GPUI pattern:** camera movement should update a transform, a spatial query, and layer invalidation metadata—not walk all nodes or rebuild all widget state. [Excalidraw Renderer.ts](https://github.com/excalidraw/excalidraw/blob/60b27588/packages/excalidraw/scene/Renderer.ts), [Excalidraw large-scene issue](https://github.com/excalidraw/excalidraw/issues/10512)

3. **Separate static content, interactive chrome, and DOM/widget-like overlays** — Excalidraw uses stacked canvases: `StaticCanvas` for document elements and `InteractiveCanvas` for selections, cursors, scrollbars, handles, and animations; both are memoized against only relevant app-state slices. Konva’s performance guide makes the same point with separate canvas layers, while warning not to create too many layers. Kookie Flow’s WebGL experiment keeps widgets/hidden textareas in a DOM layer while rendering entities, text, grid, edges, and selection in WebGL. **GPUI pattern:** define layers such as `grid`, `edges`, `nodes`, `node_snapshots`, `selection`, `handles`, `cursors`, and `widgets`; dirty and repaint each independently. [Excalidraw StaticCanvas](https://github.com/excalidraw/excalidraw/blob/60b27588/packages/excalidraw/components/canvases/StaticCanvas.tsx), [Excalidraw InteractiveCanvas](https://github.com/excalidraw/excalidraw/blob/60b27588/packages/excalidraw/components/canvases/InteractiveCanvas.tsx), [Konva layer management](https://konvajs.org/docs/performance/Layer_Management.html), [Kookie Flow README](https://github.com/KushagraDhawan1997/kookie-flow)

4. **Represent fixed-size zoom-independent UI explicitly** — Excalidraw draws selection borders, handles, focus points, dashes, and padding as `value / zoom`, keeping them constant in screen pixels. React Flow’s `NodeToolbar` deliberately does not scale with the viewport so toolbar content stays readable. tldraw has a “dynamic size mode” where stroke width and text size are relative to zoom so their screen-pixel size stays constant. **GPUI pattern:** give each primitive a scale policy: `WorldScaled`, `ScreenFixed(anchor_world, px_size)`, or `ZoomBucketed`; for fixed-size nodes, compute their world-space cull/hit bounds as `screen_px / zoom` or render them in a screen-space overlay anchored to a world point. [Excalidraw interactiveScene](https://github.com/excalidraw/excalidraw/blob/eb959128/packages/excalidraw/renderer/interactiveScene.ts), [React Flow NodeToolbar](https://reactflow.dev/api-reference/components/node-toolbar), [tldraw dynamic size PR](https://github.com/tldraw/tldraw/pull/3835)

5. **Use stable/effective zoom and LOD to avoid mid-gesture churn** — tldraw’s `getEfficientZoomLevel()` returns a stable zoom during camera movement for larger documents, and built-in shapes simplify at low zoom: shadows, pattern fills, draw-style strokes, text outlines, and image resolutions are reduced. Cytoscape similarly recommends hiding/minimizing labels, setting `min-zoomed-font-size`, simplifying edges, lowering pixel ratio, hiding edges during viewport interactions, or using a viewport texture during pan/zoom for very large graphs. **GPUI pattern:** during camera motion, freeze expensive zoom-dependent layout at an `effective_zoom` or zoom bucket; update full-quality labels/shadows/images after camera idle. [tldraw performance docs](https://tldraw.dev/sdk-features/performance), [Cytoscape performance docs](https://github.com/cytoscape/cytoscape.js/blob/master/documentation/md/performance.md)

6. **Cache geometry and rendered snapshots, but budget and invalidate them carefully** — tldraw caches shape geometry and advises using `editor.getShapeGeometry()` instead of recomputing shape utilities directly; Excalidraw memoizes renderable sets and caches link icons in small canvases keyed by zoom. PixiJS `cacheAsTexture()` turns mostly-static containers into a single texture but warns about GPU-memory cost, very large textures, sparse content, and frequent recaching; Konva’s shape caching uses the same “draw complex once, reuse image” principle. **GPUI pattern:** cache node body snapshots, text layout, shape bounds, hit regions, edge tessellation, and image mip/LOD variants keyed by content version + theme + zoom bucket; invalidate by version and enforce a memory budget/LRU. [tldraw performance docs](https://tldraw.dev/sdk-features/performance), [Excalidraw staticScene](https://github.com/excalidraw/excalidraw/blob/60b27588/packages/excalidraw/renderer/staticScene.ts), [PixiJS cache-as-texture](https://pixijs.com/8.x/guides/components/scene-objects/container/cache-as-texture), [Konva shape caching](https://konvajs.org/docs/performance/Shape_Caching.html)

7. **Batch data updates, invalidations, frame scheduling, and draw calls** — tldraw batches store updates (`updateShapes([...])`, `editor.run(...)`) so observers receive one notification; Cytoscape recommends `cy.batch()` for many modifications; Excalidraw throttles static scene renders to animation frames; Konva’s `batchDraw()` coalesces many mousemove updates into browser-limited frame draws. On GPU paths, PixiJS emphasizes grouping by object/texture/blend mode to preserve batching, and Kookie Flow reports instanced entities in one draw call with batched GPU edge geometry. **GPUI pattern:** expose scene transactions, collect invalidation ranges, schedule one paint per GPUI frame using latest input state, and render by layer/material with instanced buffers for rectangles, ports, labels, and edge segments. [tldraw performance docs](https://tldraw.dev/sdk-features/performance), [Cytoscape performance docs](https://github.com/cytoscape/cytoscape.js/blob/master/documentation/md/performance.md), [Excalidraw staticScene throttling](https://github.com/excalidraw/excalidraw/blob/60b27588/packages/excalidraw/renderer/staticScene.ts), [Konva batchDraw](https://konvajs.org/docs/performance/Batch_Draw.html), [PixiJS performance tips](https://pixijs.com/8.x/guides/concepts/performance-tips), [Kookie Flow README](https://github.com/KushagraDhawan1997/kookie-flow)

8. **Isolate state so dragging one node does not wake the world** — React Flow’s perf docs warn against components subscribing to the whole `nodes`/`edges` arrays; its `NodeRenderer` subscribes only to visible node IDs, while `NodeWrapper` subscribes to one node. A React Flow performance PR reports that avoiding rerender of the whole renderer prevented a 500-node `nodes.map()` loop from costing ~20–40ms under CPU throttling, making dragging 2–3× cheaper. tldraw’s signal system similarly updates only components that depend on changed data. **GPUI pattern:** separate model store, camera store, selection store, and transient interaction store; use per-node/per-edge versioned records and derived ID sets; never let `drag_position` changes rebuild the whole scene tree. [React Flow performance docs](https://reactflow.dev/learn/advanced-use/performance), [React Flow NodeRenderer source](https://github.com/xyflow/xyflow/blob/main/packages/react/src/container/NodeRenderer/index.tsx), [React Flow perf PR](https://github.com/xyflow/xyflow/pull/3668), [tldraw performance docs](https://tldraw.dev/sdk-features/performance)

9. **Treat edges as a separate expensive subsystem** — React Flow’s edge visibility test computes the bounding box spanning source and target nodes and intersects it with the viewport; Cytoscape notes that edges, bezier curves, arrows, dashed strokes, and semitransparency are especially expensive, and suggests hiding edges during pan/zoom for very large graphs. **GPUI pattern:** maintain edge bounds/routes separately from node widgets, cull by expanded edge boxes, keep selected/hovered/connecting edges alive, render simplified straight/low-alpha edges while panning, and run route recomputation/tessellation off the interaction path. [xyflow edge visibility source](https://github.com/xyflow/xyflow/blob/main/packages/system/src/utils/edges/general.ts), [Cytoscape performance docs](https://github.com/cytoscape/cytoscape.js/blob/master/documentation/md/performance.md)

10. **Move heavyweight rendering/tessellation off the UI path where possible** — MDN documents `OffscreenCanvas` transfer to workers and worker `requestAnimationFrame`; the `react-infinite-canvas` project is built around OffscreenCanvas + Web Workers, spatial grid indexing, batched rendering, and LOD. Native GPUI does not need browser workers, but the pattern maps cleanly to Rust: route edges, tessellate paths, rasterize snapshots, decode images, and update spatial indexes on worker threads, then atomically publish immutable render packets to the UI/render thread. [MDN OffscreenCanvas](https://developer.mozilla.org/en-US/docs/Web/API/OffscreenCanvas), [react-infinite-canvas README](https://github.com/tu2-atmanand/infinit-canvas)

11. **Instrument frame time and visibility from day one** — tldraw exposes `editor.performance` events with interaction/camera frame stats including FPS, p95/p99 frame times, visible/culled shape counts, shape count, viewport size, and zoom; it also wires events into browser performance marks. Excalidraw has a debug utility that logs per-frame average cost as a percentage of the 16.67ms frame budget. Cytoscape exposes renderer `debug`/`showFps` options. **GPUI pattern:** add a zero-overhead-when-disabled `PerfManager` with frame-time histograms, input-to-present latency, cull query time, visible/culled counts, draw calls/batches, cache hit/miss/bytes, layout/tessellation time, and per-layer paint costs. [tldraw performance docs](https://tldraw.dev/sdk-features/performance), [Excalidraw Debug](https://github.com/excalidraw/excalidraw/blob/2b0e4c96/packages/common/debug.ts), [Cytoscape renderer defaults](https://github.com/cytoscape/cytoscape.js/blob/unstable/src/core/renderer.mjs)

12. **Prefer Canvas/WebGL/Rust-native primitives for large-scale content, not per-node retained widgets** — Canvas/WebGL projects converge on batching, culling, spatial indexing, SDF/MSDF text, and shader grids. The infinite-canvas tutorial explicitly targets draw-call reduction with culling and batching, RBush picking, SDF shapes, GPU text/Bezier rendering, and tile-based ideas; MSDF text is recommended for sharper text at varying sizes. **GPUI pattern:** reserve GPUI child views/widgets for selected/open/editor nodes; render the large background population as custom primitives or snapshots, with text via cached layout/atlas/MSDF where GPUI text costs are too high. [Infinite Canvas Tutorial README](https://github.com/antvis/infinite-canvas-tutorial), [MSDF text example](https://infinitecanvas.cc/example/msdf-text.html), [PixiJS culler plugin](https://pixijs.com/8.x/guides/components/application/culler-plugin)

## Gaps
- No native GPUI/Rust infinite-canvas benchmark directly maps these web techniques to GPUI; validate with a prototype measuring 1k/5k/10k nodes, edge counts, text density, and selected-node widget overlays.
- Kookie Flow and react-infinite-canvas README FPS claims are not independently benchmarked here; treat them as architecture examples, not targets.
- Excalidraw spatial-index/dirty-region improvements are discussed in issues but not confirmed as merged in the inspected source.
- Need local decisions on whether fixed-size “nodes” means entire node cards stay screen-fixed or only labels/handles/toolbars do; culling and edge routing differ significantly.
