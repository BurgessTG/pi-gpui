# Research: GPUI/Zed and canvas-editor performance patterns for Pi Workspaces

## Summary
Pi Workspaces should improve performance by reducing invalidation scope, layout work, and rendered item counts—not by lowering steady-state visual quality. The strongest pattern is: stable GPUI `Entity` boundaries + cached views for unchanged panels, virtualized/cullable lists/canvas regions, cached text/geometry/layout, batched state updates, and opt-in instrumentation that tracks dirty reasons and frame costs.

## Findings
1. **Use GPUI cached view islands for stable workspace regions** — `AnyView::cached` can recycle a view’s previous layout and paint when `Context::notify` has not dirtied that entity; the source shows reuse is also gated by unchanged bounds, content mask, and text style, and is bypassed by `Window::refresh` or dirty views. For Pi: make columns, sidebars, markdown panes, minimap/canvas overlays, and expensive node rows separate `Entity`s, render them through cached `AnyView`s, and notify only the entity whose data changed. Risk: broad `window.refresh()` / `cx.refresh_windows()` erases cache wins, and inspector picking disables this cache path. [AnyView docs](https://docs.rs/gpui/latest/gpui/struct.AnyView.html), [GPUI view source](https://docs.rs/gpui/latest/src/gpui/view.rs.html), [Zed PR on reducing refresh](https://github.com/zed-industries/zed/pull/25009)
2. **Keep animations scoped and self-invalidating** — GPUI’s `request_animation_frame()` redraws the next frame; if called inside a view it notifies that view next frame, otherwise it refreshes the whole window. `AnimationElement` schedules the next frame only while the animation is unfinished. For Pi: host continuous animations in small child entities/elements; animate transform/opacity-like leaf properties where possible; stop RAF loops when hidden/idle/unfocused; and avoid attaching animation ticks to root workspace state. Risk: a spinner or hover animation outside a view boundary can force whole-window redraws and defeat cached views. [Window docs](https://docs.rs/gpui/0.2.2/gpui/struct.Window.html), [Animation source](https://docs.rs/gpui/latest/src/gpui/elements/animation.rs.html)
3. **Virtualize every long list; choose the list primitive by row geometry** — GPUI `uniform_list` renders only the visible range and avoids full Taffy layout by measuring one item, but it only works for uniform-height rows. `gpui-component` `VirtualList` supports variable item sizes with an `Rc<Vec<Size<Pixels>>>` and visible-range rendering. For Pi: use `uniform_list` for files, logs, tabs, search results, and homogeneous node rows; use variable-size virtualization for markdown blocks, chat/agent messages, timeline cards, and tables with dynamic rows; precompute item sizes and keep `ScrollHandle`s in entity state. Risk: gpui-component’s current visible-range scan walks item sizes linearly, so very large variable lists may need prefix sums + binary search. [GPUI uniform_list source](https://github.com/zed-industries/zed/blob/main/crates/gpui/src/elements/uniform_list.rs), [gpui-component VirtualList docs](https://longbridge.github.io/gpui-component/docs/components/virtual-list), [VirtualList source](https://docs.rs/gpui-component/latest/src/gpui_component/virtual_list.rs.html)
4. **Treat markdown as parsed/cached blocks, not a monolithic render tree** — Zed parses markdown off the foreground thread, coalesces reparses with `should_reparse`, keeps existing content visible while a parse is pending, and retains code-block scroll handles by stable source offsets. GPUI text layout caches line and wrapped-line layouts across current/previous frames and also has hash-keyed layout paths to avoid materializing text on cache hits; `TextLayout` reuses measured size when wrap width is unchanged and truncation is not active. For Pi: parse markdown in background, key rendered blocks by source hash + style + wrap width, virtualize by block, retain per-block scroll/selection state, and invalidate only changed blocks. Risk: drag scrolling or large jumps can miss previous-frame text caches; a Zed/gpui-component discussion measured `layout_wrapped_line` as a major cost for 500–1000 small text cells. [Markdown source](https://github.com/zed-industries/zed/blob/79b9cae2/crates/markdown/src/markdown.rs), [Line layout cache source](https://github.com/zed-industries/zed/blob/79b9cae2/crates/gpui/src/text_system/line_layout.rs), [Text element source](https://github.com/zed-industries/zed/blob/79b9cae2/crates/gpui/src/elements/text.rs), [Text performance discussion](https://github.com/zed-industries/zed/discussions/24260)
5. **Optimize counts and cache misses, not text/image quality** — Zed’s GPUI renderer is designed for 8.33ms 120Hz frames using GPU primitives, native OS text shaping/rasterization, shaped-text caches, and a glyph atlas; this preserves native text quality while amortizing expensive shaping/rasterization. For Pi: keep full-quality text, antialiasing, icons, and shadows in steady state; improve performance by reducing how many glyphs/elements are laid out/painted and by increasing cache hit rates. [Zed GPUI rendering blog](https://zed.dev/blog/videogame), [WindowTextSystem docs](https://docs.rs/gpui/latest/gpui/struct.WindowTextSystem.html)
6. **Apply infinite-canvas practices: cull by viewport, cache geometry, batch mutations** — tldraw keeps large canvases smooth using viewport culling, reactive signals, batched store updates, debounced zoom, geometry caching, and LOD for details that are invisible at low zoom. For Pi Workspaces: maintain a scene store with derived visible nodes/edges, cached node bounds and connector geometry, batched graph updates, and a stable zoom value for zoom-dependent stroke/text calculations during pan/zoom. Risk: disabling culling should be rare—only for measured DOM/content, off-bounds glows, or active animations. [tldraw performance docs](https://tldraw.dev/sdk-features/performance), [tldraw culling docs](https://tldraw.dev/sdk-features/culling)
7. **Use a spatial index for node hit-testing, culling, selection, and edge queries** — tldraw’s `SpatialIndexManager` uses RBush/R-tree indexing for O(log n)-class spatial queries, bulk-loads per page, incrementally updates from store history, and exposes bounds and point queries; results are unordered and must be combined with z-order if draw/hit order matters. For Pi: introduce an R-tree or grid for node bounds, edge bounds, marquee selection, visible-set queries, minimap queries, and connector routing candidates. Risk: computed bounds for groups/arrows require dependency-aware invalidation, and spatial results need a separate z-order pass. [SpatialIndexManager source](https://github.com/tldraw/tldraw/blob/f69220b8/packages/editor/src/lib/editor/managers/SpatialIndexManager/SpatialIndexManager.ts)
8. **Layer canvases/scene primitives by update frequency** — Konva documents that each layer is a separate canvas, allowing only changed layers to refresh, but warns that too many layers add overhead; it recommends non-listening static layers and moving dragged shapes to a drag layer. GPUI also exposes paint-time layers for batching non-overlapping geometry of the same draw order. For Pi: separate static grid/background, edges, nodes, selection/handles, drag ghost, and transient animation overlays; keep the layer count small; disable hit-testing/event processing for layers that never receive input. [Konva performance tips](https://konvajs.org/docs/performance/All_Performance_Tips.html), [Window layer docs](https://docs.rs/gpui/0.2.2/gpui/struct.Window.html)
9. **Avoid global graph/list dependencies in UI components** — React Flow warns against components reading the full `nodes`/`edges` arrays because drag/pan/zoom updates trigger broad rerenders; it recommends separate state for selected IDs, memoized components/functions, collapsing large trees, and simplifying expensive styles/animations. For Pi: store selection, hover, viewport, dirty node IDs, and visible IDs as narrow derived state; make node entities subscribe only to their own model + small shared style/camera state. Risk: a single top-level graph state dependency can dirty all node views and nullify GPUI caching. [React Flow performance docs](https://reactflow.dev/learn/advanced-use/performance)
10. **Use transient interaction shortcuts carefully if “full quality” is required** — Cytoscape notes labels, edge arrows, dashed edges, compound nodes, and high pixel ratio are expensive; it offers options like hiding edges during viewport interaction or texture-caching pan/zoom. PixiJS similarly advises batching draw order, culling when GPU-bound, avoiding per-frame text changes, and disabling event traversal for non-interactive children. For Pi: prefer full-quality at rest; if needed, use temporary interaction-only optimizations such as freezing edge labels or reusing a texture during rapid pan/zoom, then restore full quality on idle. Risk: permanent LOD, low pixel ratio, or hidden labels may violate the product requirement unless gated by zoom/interaction and validated visually. [Cytoscape.js performance docs](https://github.com/cytoscape/cytoscape.js/blob/master/documentation/md/performance.md), [PixiJS performance tips](https://pixijs.com/8.x/guides/concepts/performance-tips)
11. **Distinguish dirty redraws from high-refresh presentation** — Zed’s 120fps work showed that display timing and GPU submission can matter even when CPU frame time is low; it used HUDs/Instruments and frame-duration measurement to diagnose jank. A later Zed issue highlights the battery/GPU risk of re-presenting unchanged content at full rate after irrelevant input. For Pi: track why each frame exists (`data_changed`, `layout_changed`, `animation_tick`, `scroll`, `present_only`), avoid updating the dirty timestamp for irrelevant input, and skip expensive scene rebuilds when presenting unchanged content for refresh-rate responsiveness. [Zed 120fps blog](https://zed.dev/blog/120fps), [Unchanged-present GPU-load issue](https://github.com/zed-industries/zed/issues/32588)
12. **Make performance instrumentation opt-in but first-class** — Zed recommends `samply` flamecharts, Tracy tracing via `ZTRACING=1 cargo r --features tracy --release`, foreground/background task profiling through `zed open performance profiler`, and `zlog::time!(...).warn_if_gt(...)`; tldraw exposes `editor.performance` events with FPS, p95 frame time, and interaction/camera lifecycle events. For Pi: add an opt-in perf overlay/log with frame p50/p95/max, layout/prepaint/paint times, entity render counts, cached-view hit/miss counts, virtual visible ranges, text layout hit/miss counts, culled/visible shape counts, spatial-index query/update timing, animation count, and dirty reasons. Risk: FPS counters/graphics profilers can perturb performance or crash, so keep metrics low-overhead and disabled by default. [Zed performance docs](https://github.com/zed-industries/zed/blob/2aa36660/docs/src/performance.md), [tldraw performance events](https://tldraw.dev/sdk-features/performance), [Zed graphics profiler revert context](https://github.com/zed-industries/zed/pull/7607)

## Gaps
- Need verify exact GPUI/gpui-component versions used by Pi against `Cargo.lock`; APIs and cache behavior are moving.
- No Pi Workspace benchmark exists yet; recommended next step is an opt-in perf overlay plus scripted scenarios: 10k nodes, 100k edges, long markdown, rapid pan/zoom, drag selection, and concurrent agent output.
- tldraw docs mention both all-shape culling iteration and a newer spatial-index manager; verify the released tldraw version before treating the R-tree behavior as universal.
- gpui-component VirtualList may need a local fork/PR for binary-search visible ranges if Pi uses very large variable-height lists.
- Markdown block virtualization needs a small prototype to validate selection/source mapping, code-block horizontal scroll, images, and mermaid/custom blocks under partial rendering.
